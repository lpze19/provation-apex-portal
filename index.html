What is Provation Apex?

Endoscopy Documentation,
Reimagined for DHR Health

Provation Apex is DHR Health's enterprise endoscopy reporting and documentation platform. It integrates directly with the EMR, enabling real-time procedure documentation, automatic charge capture, and seamless patient record completion — from the moment a case opens to the physician's final signature.

🏥

EMR Integration

Patient data flows automatically from registration through ADT/HL7 interfaces — no manual entry required.

📸

Image Capture

Scope images are captured directly into the case record the moment the scope and physician are assigned.

✍️

Physician Signature

Physicians sign cases electronically. The completed document is automatically sent to the patient's chart.

💰

Charge Capture

Procedure charges are captured within the workflow, reducing billing lag and documentation gaps.

1a
Clinical Workflow

Admission / Registration

The admission and registration process is the entry point for all Provation Apex cases. Patient identity is confirmed, demographics are verified, and the encounter is created in the EMR. ADT HL7 messages automatically populate Provation Apex — eliminating duplicate data entry.

01
Patient Arrival

Front desk confirms patient identity, verifies insurance, and initiates pre-registration in the scheduling system.

02
ADT Message Triggered

An HL7 ADT (Admit/Discharge/Transfer) message is generated and transmitted to Provation Apex via the interface engine.

03
Case Pre-Population

Provation Apex receives the ADT message and pre-populates the case with patient demographics, MRN, encounter number, and procedure details.

04
Registration Confirmation

Clinical staff confirms the case in Provation Apex, verifying all pre-populated data before the patient enters the procedure area.

Key Interface
HL7 ADT v2.x
Trigger Event
A01 — Admit Patient
Data Populated
MRN, Name, DOB, Insurance, Encounter #
Manual Entry Required
None — fully automated via interface
1b
Clinical Workflow

Scheduling

Procedure scheduling is managed through the hospital scheduling system and communicated to Provation Apex via SIU (Schedule Information Unsolicited) HL7 messages. This ensures the procedure list in Provation Apex always reflects the current schedule without manual synchronization.

01
Procedure Scheduled

Scheduler books the procedure in the hospital scheduling system, selecting procedure type, physician, room, and time slot.

02
SIU Message Sent

An SIU S12 (New Appointment) message is transmitted to Provation Apex, creating a pending case in the worklist.

03
Worklist Updated

The case appears in the Provation Apex daily worklist, visible to clinical staff and the assigned physician.

04
Schedule Changes

Cancellations and reschedules trigger SIU S15/S16 messages, automatically updating or removing the case from the worklist.

Interface Type
HL7 SIU v2.x
Message Events
S12 New · S13 Reschedule · S15 Cancel
Worklist Visibility
Real-time — no manual refresh needed
Scheduling System
Hospital Scheduling / EMR
1c
Clinical Workflow

Orders

Procedure orders originate in the EMR and are transmitted to Provation Apex via HL7 ORM messages. Orders define the procedure type, clinical indications, and any special instructions — forming the clinical basis for the Provation Apex case documentation.

01
Order Placed in EMR

Physician places a procedure order in the EMR (e.g., colonoscopy, upper endoscopy). The order includes diagnosis codes, clinical indications, and urgency.

02
ORM Message Transmitted

An HL7 ORM O01 message is sent to Provation Apex, linking the order to the scheduled case.

03
Case Enrichment

Provation Apex populates the case with procedure codes, indications, and clinical context from the order — pre-filling documentation templates.

04
Order Acknowledgment

Provation Apex sends an ORR acknowledgment back to the EMR confirming receipt and case linkage.

Interface Type
HL7 ORM v2.x
Message Event
O01 — General Order
Data Included
CPT codes, ICD-10, Indications, Urgency
Acknowledgment
ORR — Order Response
1d
Clinical Workflow

Charting / Documentation

Real-time charting occurs throughout the procedure. Provation Apex provides structured documentation templates for each anatomical section, capturing findings, images, measurements, and clinical observations as the procedure progresses.

01
Case Opened

Clinical staff opens the case in Provation Apex. The structured documentation template loads based on the procedure type.

02
Real-Time Image Capture

Scope images are captured directly into the case record via the Olympus or Pentax imaging interface. Images are automatically tagged to the current anatomical section.

03
Findings Documentation

Clinical staff documents findings at each anatomical section using structured templates — polyp size, location, morphology, interventions performed.

04
Specimen Labels

If specimens are collected, Provation Apex generates specimen labels with patient ID, collection site, and procedure details for pathology submission.

Template Type
Structured — procedure-specific
Image Systems
Olympus CV-180/190, Pentax IMAGINA
Specimen Labels
Auto-generated with pathology data
Offline Support
Yes — local cache with sync on reconnect
1e
Clinical Workflow

Physician Note

The physician note is the clinical narrative of the procedure — findings, impressions, recommendations, and follow-up instructions. Provation Apex auto-generates a structured note from the documented data, which the physician reviews and finalizes before signing.

01
Auto-Generated Draft

Upon scope withdrawal, Provation Apex assembles a structured procedure note from all documented findings, images, and interventions.

02
Physician Review

The physician reviews the auto-generated note, adding impressions, recommendations, follow-up instructions, and any additional clinical narrative.

03
Note Finalization

The physician finalizes the note content. At this stage the note is complete but not yet legally signed.

04
Electronic Signature

The physician applies their electronic signature. The note is timestamped, locked, and transmitted to the EMR chart automatically.

Note Generation
Auto-assembled from structured data
Signature Type
Electronic — credential-bound
Post-Signature
Locked — no alterations permitted
Distribution
Auto-sent to EMR chart via ORU message
1f
Clinical Workflow

Charges

Charge capture is embedded in the Provation Apex documentation workflow. Procedure codes are automatically suggested based on documented findings and interventions, reducing billing lag and ensuring accurate charge submission.

01
Auto-Suggested Codes

Based on documented procedures and findings, Provation Apex suggests CPT and ICD-10 codes for the encounter.

02
Clinical Review

Clinical staff or the physician reviews and confirms the suggested codes, adding or modifying as clinically appropriate.

03
Charge Submission

Upon case completion, charges are exported to the billing system via the configured interface, eliminating manual charge entry.

04
Audit Trail

All charge activity is logged with timestamps and user credentials, providing a complete audit trail for compliance and billing review.

Code Types
CPT, ICD-10-CM, HCPCS
Trigger

Export Method
Billing interface / HL7 DFT
Audit Log

1g
Clinical Workflow

Transfer

Patient transfers — between units, rooms, or facilities — are communicated to Provation Apex via ADT A02 transfer messages. The case record is updated automatically to reflect the patient's new location and care team.

01
Transfer Initiated

Clinical staff initiates a transfer in the EMR, specifying the destination unit, room, or facility.

02
ADT A02 Message

An HL7 ADT A02 (Transfer Patient) message is transmitted to Provation Apex, updating the patient location in the case record.

03
Case Record Updated

The Provation Apex case reflects the new location. Any pending documentation or unsigned notes are flagged for the receiving care team.

04
Continuity of Care

All case documentation, images, and notes remain accessible to the receiving team. No data is lost or duplicated during transfer.

HL7 Event
ADT A02 — Transfer Patient
Data Updated
Location, Room, Attending Physician
Documentation
Fully preserved — no re-entry needed
Pending Items
Flagged for receiving team review
1h
Clinical Workflow

Discharge

Discharge closes the clinical encounter in Provation Apex. An ADT A03 message triggers final case reconciliation — ensuring all documentation is complete, all charges are captured, and the procedure note is signed and transmitted to the patient's permanent chart.

01
Discharge Initiated

Clinical staff initiates discharge in the EMR after confirming the patient is clinically ready and all documentation is complete.

02
ADT A03 Message

An HL7 ADT A03 (Discharge Patient) message is sent to Provation Apex, triggering final case reconciliation.

03
Documentation Check

Provation Apex verifies that the physician note is signed, charges are captured, and all required fields are complete. Incomplete items are flagged.

04
Case Closed

The case is closed and archived. The completed procedure note is permanently stored in the patient's EMR chart and available for future reference.

HL7 Event
ADT A03 — Discharge Patient
Validation
Signature, charges, required fields
Final Output
Signed note → EMR chart (ORU)
Case Status
Closed & Archived

Application Purpose

Provation Apex is an enterprise endoscopy reporting and documentation platform designed to streamline the complete clinical documentation lifecycle for gastrointestinal procedures. It replaces paper-based and fragmented documentation workflows with a single, integrated system.

At DHR Health, Provation Apex serves as the system of record for all endoscopy procedures — capturing real-time clinical data, images, and physician notes, then automatically distributing completed documentation to the EMR, billing, and downstream systems.

Authentication

Provation Apex at DHR Health uses Azure Active Directory (Azure AD) for Single Sign-On (SSO) authentication. Users log in with their standard DHR Health network credentials — no separate Provation username or password is required.

01
User Launches Provation Apex

User navigates to the Provation Apex URL or launches the application from their workstation.

02
Azure AD Redirect

The application redirects to the Microsoft Azure AD login page (login.microsoftonline.com) for authentication.

03
DHR Credentials

User enters their standard DHR Health network username and password. MFA may be required per DHR security policy.

04
Token Issued

Azure AD issues an OAuth 2.0 / SAML token. Provation Apex validates the token and grants access based on the user's assigned role.
